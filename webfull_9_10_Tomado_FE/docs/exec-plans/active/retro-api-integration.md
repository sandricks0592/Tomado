# Retro API 연동

## 목표

- Retro 페이지를 테스트용 로컬 데이터 기반에서 실제 API 기준으로 전환한다.
- 생성/수정/삭제 이후 목록과 선택 상태를 실제 데이터 기준으로 정리한다.

## 범위

- `src/pages/Retro.tsx`
- `src/api/generated/retro-logs/retro-logs.ts`

## 현재 상태

- 회고 생성/수정/삭제 mutation은 이미 연결되어 있다.
- 목록 영역은 아직 `testRetroArr` 로컬 배열을 사용한다.
- 삭제는 카테고리 타입 선택 + 지연 삭제 + 토스트 undo 패턴을 사용한다.
- 생성 후 목록 재조회가 없어 현재 선택 상태와 목록 동기화가 완전하지 않다.

## 작업 단계

1. 날짜별 회고 목록 조회 API 연결
2. 생성/수정/삭제 후 재조회 또는 캐시 동기화 기준 확정
3. 검색 흐름 필요 여부 검토
4. 페이지 상태와 API 동기화 로직 분리 여부 검토
5. 관련 design doc / product spec 갱신

## 주요 결정 포인트

- 회고 목록을 날짜 단위로 묶어 받을지, 프론트에서 그룹핑할지
- 카테고리별 삭제 후 선택 상태를 어떻게 재설정할지

## 리스크

- 목록 재조회 기준이 없으면 생성/삭제 후 선택 상태가 꼬일 수 있다.
- 카테고리별 삭제와 autosave가 겹치면 상태 관리가 복잡해질 수 있다.

## 완료 조건

- 회고 목록이 실제 API 데이터로 렌더링된다.
- 생성/수정/삭제 후 목록과 선택 상태가 일관되게 동기화된다.
- 테스트용 로컬 회고 데이터가 제거된다.
