# Todo Flow

이 문서는 Tomado의 Todo 기능이 사용자 관점에서 어떻게 동작해야 하는지 정리합니다.

구현 구조와 내부 책임은 [Todo Domain](../design-docs/todo.md)를 기준으로 확인합니다.

## 목표

- 사용자가 날짜별로 할 일을 추가, 수정, 체크, 삭제, 이동, 정렬할 수 있어야 한다.
- 같은 날짜 안에서는 사용자가 정한 순서를 유지해야 한다.
- 삭제 이후에는 짧은 시간 동안 되돌릴 수 있어야 한다.

## 핵심 사용자 흐름

### 1. 할 일 추가

1. 사용자는 입력창에 할 일을 입력한다.
2. 글자 수 제한을 넘기면 입력 제한과 오류 안내가 동작한다.
3. Enter 또는 추가 액션으로 새 Todo를 생성한다.
4. 생성 직후에는 낙관적 업데이트로 목록에 즉시 반영된다.
5. 서버 응답이 오면 실제 Todo 데이터로 동기화된다.

### 2. 체크 / 완료 상태 변경

1. 사용자는 체크박스로 Todo의 완료 여부를 바꾼다.
2. 완료 상태는 프론트 캐시에 먼저 반영되어 즉시 시각적으로 바뀐다.
3. 완료된 항목은 시각적으로 구분되며, 이 상태를 기준으로 삭제 가능 여부도 즉시 바뀐다.

### 3. 라벨 수정

1. 사용자는 Todo 라벨을 직접 수정할 수 있다.
2. 입력 중에는 로컬 draft만 변경된다.
3. blur 또는 Enter 시점에만 실제 수정 요청이 전송된다.
4. 빈 값이면 기존 라벨로 복구된다.

### 4. 정렬 변경

1. 사용자는 drag handle로 Todo 순서를 바꿀 수 있다.
2. 같은 날짜 안에서만 순서를 조정한다.
3. 순서가 바뀌면 해당 날짜 그룹의 order가 다시 계산된다.

### 5. 날짜 이동

1. 사용자는 more 메뉴에서 날짜 이동을 선택한다.
2. 날짜 선택 모달에서 이동할 날짜를 고른다.
3. 이동 후에는 대상 날짜 그룹의 마지막 순서 뒤에 붙고, 원래 날짜 그룹은 order가 다시 정리된다.

### 6. 삭제 및 복구

1. 사용자는 more 메뉴에서 삭제를 선택한다.
2. 항목은 목록에서 제거된다.
3. 삭제 후 약 3초 동안 toast에서 취소를 누를 수 있다.
4. 취소하지 않으면 그 시점에 실제 삭제 요청이 전송된다.

## 데이터 규칙

- Todo는 `assignedDate`를 기준으로 날짜별 그룹을 가진다.
- 각 날짜 그룹 안에서는 `order` 값으로 정렬된다.
- 추가, 이동, 삭제, 복구, 정렬 변경은 모두 날짜별 order 규칙을 유지해야 한다.

## 표시 규칙

- 기본 Todo 패널은 날짜별 목록을 보여준다.
- focus mode 안에서는 같은 Todo 패널을 보조 패널로 사용한다.
- more 메뉴는 기본 톤에서만 노출될 수 있다.
- 초기 목록 조회가 지연될 때는 Todo 행 스켈레톤을 먼저 보여준다.

## 예외 및 실패 처리

- 현재 구현은 Todo API와 React Query를 source of truth로 사용한다.
- 초기 조회 중에는 스켈레톤을 먼저 노출하고, 이미 목록이 있는 상태의 background refetch에서는 기존 목록을 유지한다.
- 완료 토글은 프론트 캐시에 먼저 반영하고, 실패하면 이전 상태로 롤백한다.
- 삭제는 toast 노출 시간 동안만 지연 처리되고, 이후 실제 삭제 요청이 전송된다.
- 날짜 이동과 정렬 변경은 같은 날짜 그룹 order 규칙을 깨지 않도록 처리해야 한다.

## 현재 구현 메모

- 현재 Todo는 Todo API와 React Query 기반으로 동작한다.
- `TodoPanel`이 입력, DnD 정렬, 날짜 이동 모달 연결까지 함께 담당한다.
- 추가, 완료 토글, 정렬 변경은 사용자 체감 지연을 줄이기 위해 프론트 반영을 먼저 수행한다.
- 삭제는 toast undo를 위해 지연 삭제 방식으로 동작한다.
