# Plans

이 문서는 실행 계획 문서를 어떻게 만들고 운영할지 정리합니다.

실제 계획 문서는 `docs/exec-plans/` 아래에 두고, 이 문서는 운영 규칙을 설명합니다.

## Directory Structure

```txt
docs/exec-plans/
├── active/
├── completed/
└── tech-debt-tracker.md
```

## Purpose

- 큰 작업을 코드 밖에서도 추적 가능하게 유지합니다.
- 여러 번에 나눠 진행하는 리팩터링의 맥락을 저장합니다.
- 결정 이유와 남은 리스크를 기록합니다.
- 에이전트와 사람이 같은 계획 문서를 기준으로 협업할 수 있게 합니다.

## When To Create A Plan

아래에 해당하면 `active/`에 계획 문서를 만듭니다.

- 여러 파일에 걸친 리팩터링
- 한 번에 끝나지 않는 구조 변경
- 순서가 중요한 작업
- 중간 결정 로그가 필요한 작업
- PR 하나 이상에 걸쳐 이어질 가능성이 있는 작업

반대로 아래는 굳이 별도 계획 문서 없이 진행할 수 있습니다.

- 단일 파일 수정
- 작은 버그 수정
- 의사결정이 거의 필요 없는 단순 변경

## Active Plan Template

권장 구성:

```md
# 작업명

## 목표

## 범위

## 현재 상태

## 작업 단계

## 결정 사항

## 리스크

## 완료 조건
```

## Completed Plans

- 작업이 끝나면 `active/`에서 `completed/`로 이동합니다.
- 완료 후에는 결과와 후속 과제만 간단히 남깁니다.
- 완료 문서는 "왜 이렇게 바뀌었는지"를 추적하는 기록으로 유지합니다.

## Tech Debt

- 개별 active plan으로 바로 만들기 어려운 부채는 `tech-debt-tracker.md`에 기록합니다.
- 조건:
    - 아직 바로 착수하지 않을 작업
    - 구조상 문제는 있지만 우선순위가 낮은 작업
    - 후속 리팩터링 후보

## Writing Rules

- 계획 문서는 구현 디테일보다는 작업 흐름과 의사결정 중심으로 씁니다.
- 코드와 문서가 어긋나면 문서를 즉시 갱신합니다.
- 이미 끝난 사실을 `active/`에 오래 남겨두지 않습니다.
- 너무 큰 마스터 플랜 하나보다, 작업 단위별 문서를 선호합니다.

## Current Usage

- Timer 구조 정리 같은 다단계 리팩터링은 계획 문서 대상입니다.
- Todo 구조 개선도 계획 문서로 관리하기 적합합니다.
